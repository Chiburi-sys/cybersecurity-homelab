# Wazuh Detection Engineering: Linux Crash & Hardware Failure Rules

**Author:** Stacy  
**Date:** 2026-03-21  
**SIEM Platform:** Wazuh (OpenSearch backend)  
**Objective:** Enhance visibility into critical endpoint unreliability events (kernel panics, unexpected reboots, hardware failures) by authoring custom XML decoders and rules.

---

## 1. The Challenge

By default, Wazuh comes with comprehensive out-of-the-box alerting for Windows event logs regarding unexpected shutdowns (e.g., Event ID 41, 6008). However, it lacks native, high-fidelity alerting for bare-metal Linux crashes and hardware anomalies (e.g., thermal events, out-of-memory kills, bad memory sectors). 

As a SOC Analyst, tracking endpoint health is critical. An unresponsive system could signify a denial of service (DoS) attack, resource exhaustion malware (like cryptominers), or simply aging hardware that poses a risk to availability.

## 2. The Solution: Custom Rule Creation

To solve this, I authored a custom `local_rules.xml` file containing 9 specific rules targeting the Linux `syslog` facility. These rules parse logs generated by `systemd`, the Linux kernel, and power management utilities.

### Rule Hierarchy & Logic

All rules use the generic `<if_group>syslog</if_group>` condition, meaning they evaluate any log that Wazuh has classified as a syslog message.

#### A. Kernel Panic & Watchdog Resets (T1499: Endpoint Denial of Service)
I mapped these directly to the MITRE ATT&CK framework because an attacker causing a kernel panic or freezing the system until the hardware watchdog resets it constitutes a DoS attack.

```xml
  <!-- Kernel Panic Detected -->
  <rule id="100010" level="12">
    <if_group>syslog</if_group>
    <match>Kernel panic</match>
    <description>CRITICAL: Kernel panic detected. System crash occurred.</description>
    <mitre><id>T1499</id></mitre>
    <group>system_shutdown,kernel,</group>
  </rule>
```
*Level 12 (High) is used here because a kernel panic represents a complete loss of availability.*

#### B. Resource Exhaustion (T1499.001)
If cryptojacking malware or a run-away process consumes all RAM, the Linux OOM (Out of Memory) Killer terminates processes to save the kernel. Tracking this helps identify resource bottlenecks or malicious payloads.

```xml
  <!-- OOM Killer Activated -->
  <rule id="100014" level="10">
    <if_group>syslog</if_group>
    <match>Out of memory: Killed process|oom-kill|invoked oom-killer</match>
    <description>WARNING: OOM Killer activated. System ran out of memory and killed a process.</description>
    <mitre><id>T1499.001</id></mitre>
    <group>system_shutdown,memory,</group>
  </rule>
```

#### C. Hardware Health (MCEs and Thermal)
Monitoring Machine Check Exceptions (MCE) prevents data corruption by alerting on failing RAM or CPU cache before the system completely dies.

```xml
  <!-- Machine Check Exception (hardware failure) -->
  <rule id="100015" level="12">
    <if_group>syslog</if_group>
    <match>Machine check events logged|Hardware Error</match>
    <description>CRITICAL: Machine Check Exception (MCE). Hardware failure detected.</description>
    <group>system_shutdown,hardware,</group>
  </rule>
```

## 3. Deployment and Validation

1. **Deployment:** The rules were deployed to the Wazuh Manager at `/var/ossec/etc/rules/local_rules.xml`.
2. **Syntax Validation:** During initial deployment, the `wazuh-analysisd` daemon failed to compile due to an invalid generic tag (`<decoded_as>syslog</decoded_as>`). I identified the compilation error in `/var/ossec/logs/ossec.log`, removed the deprecated tag, and replaced it with `<if_group>`, successfully resolving the daemon crash. 
3. **Activation:** The Wazuh Manager was restarted, successfully loading `18486` total rules (including the 9 custom ones).

## 4. Business Impact

By implementing these custom rules, I expanded the SIEM's capability from purely security-focused monitoring to encompassing IT operations and endpoint reliability. This enables the SOC to proactively notify infrastructure teams of failing hardware or investigate potential resource-exhaustion attacks before they cause prolonged downtime.
